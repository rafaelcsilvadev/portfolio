# AGENTS.md

## Project overview

Personal portfolio built with Next.js (App Router), TypeScript and Tailwind CSS.
Strict **Modular MVVM** architecture with **SSR always enabled**. Interactivity
via JSX events and Server Actions — no hooks, no Zustand, no client-side state.
All text content is managed through `next-intl` translation files — no hardcoded
strings in components.

---

## ⚠️ Mandatory rule before any task

**Always read `/documents` before creating or modifying any component.**

The `/documents` folder is the single source of truth for this project. It contains:
- Components generated by Stitch (style and structure reference)
- Color guide, typography and visual design tokens
- Architecture decisions and adopted patterns
- Any other context relevant to the project

> After reading `/documents`, understand the visual and code pattern, then
> **replicate the style** when creating something new — do not copy code directly.
> If a similar component already exists in Stitch, follow its naming convention,
> spacing and JSX structure.

## Non-negotiable architectural principles

| Principle                    | Rule                                                                    |
|------------------------------|-------------------------------------------------------------------------|
| SSR always enabled           | Every component is a Server Component by default — never `"use client"` |
| Zero hooks                   | Any `use*` is forbidden — no exceptions                                 |
| Zero client-side state       | No Zustand, no useState, no context                                     |
| Interactivity via events     | `onClick`, `onChange` etc. directly in JSX — no local state             |
| Mutations via Server Actions | Every write/action with side effects goes in `viewmodel/`               |
| No hardcoded text            | Every visible string must come from a `next-intl` translation file      |

---

## Architecture: Modular MVVM

```
app/          → Next.js routes only. Zero logic, zero own components.
src/feature/  → Domain modules. Each feature is self-contained (internal MVVM).
src/core/     → Global shared code across features.
messages/     → Translation files (next-intl). One JSON per locale.
```

### MVVM layer responsibilities

| Layer       | Location                        | Responsibility                                      |
|-------------|---------------------------------|-----------------------------------------------------|
| Model       | `feature/[name]/model/`         | API response types, raw data fetching               |
| Controller  | `feature/[name]/controller/`    | Data transformation, business logic, Server Actions |
| View        | `feature/[name]/view/`          | Pure visual components, JSX events, i18n strings    |

> **Model and Controller exist only when there is an API involved.**

### Dependency rules

```
app/          can import from  → src/feature/, src/core/
src/feature/  can import from  → src/core/
src/feature/  CANNOT import from → another src/feature/
src/core/     CANNOT import from → src/feature/
```

> If two features need to share something, that something belongs in `src/core/`.

## `app/` — Routes only

- Only allowed files: `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`
- `page.tsx` imports the feature entry component and renders it — nothing more
- Every `page.tsx` must export `metadata` with `title`, `description` and `og:image`
- No `"use client"`, no logic, no own JSX

```tsx
// app/projects/page.tsx ✅ correct
import { ProjectsPage } from '@/feature/projects/view/ProjectsPage'

export const metadata = { title: 'Projetos', description: '...' }

export default function Page() {
  return <ProjectsPage />
}
```

### Model — `feature/[name]/model/`
Raw API contract. Only types and fetch calls — no transformation, no UI.

```ts
// src/feature/projects/model/projects.model.ts
export interface ProjectApiResponse {
  id: string
  title: string
  description: string
  stack: string[]
  url: string
  thumbnail: string
}

export async function fetchProjects(): Promise<ProjectApiResponse[]> {
  const res = await fetch('https://api.example.com/projects', { next: { revalidate: 3600 } })
  if (!res.ok) throw new Error('Failed to fetch projects')
  return res.json()
}
```

### Controller — `feature/[name]/controller/`
Transforms raw API data into view-ready shapes. Handles Server Actions for mutations.
Never imports components. Never uses JSX.

```ts
// src/feature/projects/controller/projects.controller.ts
import { fetchProjects, type ProjectApiResponse } from '../model/projects.model'

export interface Project {
  id: string
  title: string
  description: string
  tags: string[]
  href: string
  imageUrl: string
}

function toProject(raw: ProjectApiResponse): Project {
  return {
    id: raw.id,
    title: raw.title,
    description: raw.description,
    tags: raw.stack,
    href: raw.url,
    imageUrl: raw.thumbnail,
  }
}

export async function getProjects(): Promise<Project[]> {
  const raw = await fetchProjects()
  return raw.map(toProject)
}
```

```ts
// src/feature/contact/controller/contact.action.ts
'use server'

export async function sendMessage(formData: FormData) {
  const name = formData.get('name') as string
  const email = formData.get('email') as string
  // sending logic...
}
```

### View — `feature/[name]/view/`
Pure Server Components. Receive view-ready data from the controller.
All visible text must use `next-intl` — never hardcoded strings.

```tsx
// src/feature/projects/view/ProjectsPage.tsx ✅ correct
import { getTranslations } from 'next-intl/server'
import { getProjects } from '../controller/projects.controller'
import { ProjectCard } from './ProjectCard'

export async function ProjectsPage() {
  const t = await getTranslations('projects')
  const projects = await getProjects()

  return (
    <main>
      <h1>{t('title')}</h1>
      {projects.map((project) => (
        <ProjectCard key={project.id} {...project} />
      ))}
    </main>
  )
}
```

```tsx
// src/feature/projects/view/ProjectCard.tsx ✅ correct
import { getTranslations } from 'next-intl/server'
import type { Project } from '../controller/projects.controller'

export async function ProjectCard({ title, description, href }: Project) {
  const t = await getTranslations('projects')

  return (
    <article className="rounded-xl border p-4">
      <h2 className="text-lg font-medium">{title}</h2>
      <p className="text-sm text-muted-foreground">{description}</p>
      <a href={href}>{t('viewProject')}</a>
    </article>
  )
}
```

## i18n — next-intl

All visible text in the application must come from translation files.
**Never hardcode strings directly in components.**

### File structure

```
messages/
└── pt.json              → Portuguese (default and only locale)
```

### JSON structure — namespaced by feature

```json
{
  "common": {
    "loading": "Carregando...",
    "error": "Algo deu errado.",
    "backToHome": "Voltar ao início"
  },
  "projects": {
    "title": "Projetos",
    "viewProject": "Ver projeto",
    "noProjects": "Nenhum projeto encontrado."
  },
  "about": {
    "title": "Sobre mim",
    "bio": "..."
  },
  "contact": {
    "title": "Contato",
    "namePlaceholder": "Seu nome",
    "emailPlaceholder": "Seu e-mail",
    "submit": "Enviar mensagem",
    "successMessage": "Mensagem enviada com sucesso!"
  }
}
```

### Usage in Server Components

```ts
// Always use getTranslations (server-side) — never useTranslations (hook)
import { getTranslations } from 'next-intl/server'

const t = await getTranslations('projects')
t('title')        // → "Projetos"
t('viewProject')  // → "Ver projeto"
```

### Rules

| Rule                                                              | Reason                              |
|-------------------------------------------------------------------|-------------------------------------|
| Always use `getTranslations` from `next-intl/server`             | SSR — never the client-side hook    |
| Never use `useTranslations`                                       | It is a hook — forbidden            |
| Every new string must be added to `messages/pt.json` first       | Single source of truth for copy     |
| Namespace keys by feature (`projects.title`, `contact.submit`)   | Avoids key collisions               |
| Never duplicate strings — reuse from `common` when possible      | Consistency across the UI           |

## ❌ Hooks and client-side state — forbidden without exception

| Temptation                        | Correct solution                                           |
|-----------------------------------|------------------------------------------------------------|
| `useState` for toggle/tab         | CSS `:checked` + `<input type="checkbox">` or `<details>` |
| `useState` for form               | `<form action={serverAction}>` with `FormData`             |
| `useEffect` for data fetching     | `async` directly in the Server Component                   |
| `useEffect` for side effects      | Server Action or rethink — it is not necessary             |
| `useRef` for focus                | `autoFocus` attribute on the HTML element                  |
| `useContext` for theme            | CSS custom properties (`--color-*`) in `globals.css`       |
| `useTranslations`                 | Use `getTranslations` from `next-intl/server` instead      |
| Zustand / any store               | Data comes from the server — no client-side state          |
| `"use client"`                    | Forbidden — all components are Server Components           |

> If you cannot find a solution without hooks, **stop and ask** rather than
> silently introducing any `"use client"` directive or hook.

## Allowed interactivity

JSX events are allowed **only** for purely visual interactions with no state storage.

```tsx
// ✅ allowed — event with no state
<button onClick={() => window.print()}>Print</button>

// ✅ allowed — form submission via Server Action
<form action={sendMessage}>...</form>

// ✅ allowed — CSS class toggle, no React state
<button onClick={(e) => e.currentTarget.closest('.card')?.classList.toggle('expanded')}>
  Read more
</button>

// ❌ forbidden — stores state with a hook
const [open, setOpen] = useState(false)
<button onClick={() => setOpen(!open)}>...</button>
```

## Code conventions

### Components
- Names in **PascalCase**, each in its own folder
- **Named exports only** — never `export default` on components
- Props typed with `interface` in the file or in a local `types.ts`
- No `"use client"` — all components are Server Components
- Mark component functions as `async` when they call `getTranslations` or fetch data

### TypeScript
- `strict: true` in `tsconfig.json` — never disable strictness flags
- No `any` — use `unknown` with explicit narrowing
- `interface` for props and domain objects
- `type` for unions and aliases (`type Theme = 'light' | 'dark'`)

### Tailwind CSS
- Classes directly in JSX — no per-component `.css` files
- Use `cn()` from `@/core/utils` for conditional classes
- Mobile-first: `text-sm md:text-base lg:text-lg`
- Color tokens must align with the guide in `/documents`

### Imports and aliases
```
@/feature/[name]/...   → domain modules
@/core/...             → global shared code
```
- Always use `@/` aliases — never `../` across different zones
- Inside a feature, relative imports between layers are allowed

## Critical restrictions

| Forbidden                                          | Reason                                            |
|----------------------------------------------------|---------------------------------------------------|
| Any hook (`use*`) including `useTranslations`      | Zero client-side state — SSR always               |
| `"use client"` in any file                         | All components are Server Components              |
| Hardcoded strings in components                    | All text must live in `messages/pt.json`          |
| Zustand or any client-side store                   | Replaced by Server Actions + server data          |
| Storing state inside JSX events                    | Events may trigger actions, never store state     |
| Logic or own JSX in `app/`                        | `app/` is exclusively for routes                  |
| Feature importing from another feature             | Violates modularity — extract to `core/`          |
| `core/` importing from `feature/`                 | Violates dependency hierarchy                     |
| Domain component in `core/`                       | `core/` is generic — no feature domain knowledge  |
| `export default` on components                    | Hinders refactoring and tree-shaking              |
| Raw `<img>` or `<a>` tags                         | Use `next/image` and `next/link`                  |
| Fetching data in View layer                       | View renders only — fetch belongs in Model        |
| Transforming data in View layer                   | Transformation belongs in Controller              |
| Creating a component without reading `/documents` | Risk of visual inconsistency                      |
